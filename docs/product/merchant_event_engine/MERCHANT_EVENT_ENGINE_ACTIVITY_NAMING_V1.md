# 1. 文档定义

MERCHANT_EVENT_ENGINE_ACTIVITY_NAMING_V1，中文名为：

```text
商家活动引擎活动命名规范 V1
```

该文档用于统一 AR游伴在景区、园区、街区、商圈、商家联动活动中的命名体系。

本规范的核心判断是：

```text
AR游伴活动不是普通“优惠券活动”，也不是普通“打卡活动”，而是基于中国祝福文化的东方接福体验。
```

因此，后续对外推广、节庆活动、月度运营、商家活动、卡券权益、活动信物、完成凭证等表达，应优先使用：

```text
接福
集福
纳福
寻福
福礼
福印
福缘
福册
```

而不是默认使用：

```text
优惠券
抢券
打卡
促销
秒杀
大促
任务奖励
```

---

# 2. 命名核心原则

AR游伴的核心产品气质是：

```text
中国祝福文化
东方探索体验
真实场景中的福气收集
景区 / 园区 / 商家联动
```

一句话表达：

```text
用 AR 让游客在真实场景中遇见祝福、收集祝福、带走祝福。
```

活动命名原则：

```text
1. 优先使用“福”相关汉字，形成长期心智占位。
2. 优先使用用户能听懂的传统文化词，不使用晦涩玄学表达。
3. 活动名称应适合景区 / 园区 / 商家 / 节庆联合推广。
4. 商家权益不直接叫优惠券，应优先叫“福礼”。
5. 活动信物不直接叫奖励，应优先叫“福印”。
6. 用户参与动作不叫打卡，应优先叫“接福 / 集福 / 寻福”。
7. 后台字段可以保留 coupon / activity / reward 等技术命名，但用户端展示应使用福文化表达。
```

# 4. 活动命名分层

AR游伴活动命名建议分为四层：

# 5. 节庆型活动命名

适用场景：

```text
春节
元宵
端午
七夕
中秋
国庆
重阳
地方庙会
园区周年庆
```

推荐结构：

```text
地点 + 节日 + 接福 / 纳福 / 集福 + 会 / 节 / 游
```

推荐命名：

```text
爱企谷中秋接福会
爱企谷新春纳福节
爱企谷元宵寻福夜
爱企谷端午集福令
爱企谷七夕祈福游
爱企谷国庆迎福行
爱企谷重阳登高纳福会
```

优先推荐：

```text
中秋接福会
新春纳福节
元宵寻福夜
端午集福令
```

命名说明：

```text
接福：适合用户参与动作，亲切、直接、有仪式感。
纳福：适合春节、新年、大型节庆，有更强祝愿感。
集福：适合连续多点位、多商家、多任务活动。
寻福：适合探索感更强的园区 / 景区玩法。
```

# 6. 月度固定活动命名

适用场景：

```text
每月固定活动
园区常态运营
商家联合周
用户回访活动
```

推荐命名：

```text
爱企谷每月集福周
爱企谷月初接福日
爱企谷十五接福日
爱企谷春季纳福会
爱企谷年度万福游园会
```

建议固定节奏：

```text
每月第一周：集福周
每月十五：接福日
每季度一次：纳福季
每年一次：万福游园会
```

用户心智目标：

```text
每个月来爱企谷，都可以集一次福。
```

运营价值：

```text
形成固定活动档期
降低每次活动重新教育成本
帮助景区 / 园区形成常态化活动 IP
帮助商家形成稳定参与预期
```

# 10. 首场活动命名修正建议

原活动名：

```text
爱企谷初见寻宝节
```

建议升级为：

```text
爱企谷初见接福会
```

原因：

```text
1. “接福”比“寻宝”更符合 AR游伴的祝福文化定位。
2. “初见”保留首场样板活动的启动意味。
3. “会”比“节”更轻，适合第一场试点活动。
4. 名称更容易被游客、商家、园区理解。
5. 能和后续“集福周”“纳福节”形成体系。
```

备用名称：

```text
爱企谷初见集福周
爱企谷初见寻福令
爱企谷福印探索会
爱企谷接福游园会
```

首选：

```text
爱企谷初见接福会
```

# 15. 与现有体系关系

本命名规范不改变以下底层产品规则：

```text
活动信物仍然是活动信物，不进入主线信物合成。
数字藏品仍然是营销传播资产，不承担主线剧情。
卡券仍然是商家权益和核销对象。
活动仍然需要后台审核、发布、数据记录和复盘。
```

本命名规范只改变：

```text
用户端展示语言
活动运营命名
商家沟通表达
景区 / 园区宣传口径
```

# 16. 风险控制

## 16.1 风险：福文化表达过度玄学

控制：

```text
使用“接福、集福、纳福、福礼、福印”等大众文化词。
避免“灵验、改运、神赐、转运必成”等过度承诺词。
```

## 16.2 风险：商家误解为宗教活动

控制：

```text
强调民俗祝福文化、节庆文化、园区互动活动。
不涉及宗教承诺。
不涉及功效承诺。
```

## 16.3 风险：卡券功能被弱化

控制：

```text
后台仍保留卡券、核销、领取、库存、有效期等真实功能。
用户端将其包装为“福礼券”，但不掩盖使用规则。
```

## 16.4 风险：老文档已有“寻宝节”名称

控制：

```text
不强行覆盖老文档。
本规范作为命名升级依据。
后续新文档、用户端展示和后台展示字段优先使用“爱企谷初见接福会”。
历史文档中的“爱企谷初见寻宝节”可视为早期工作名。
```
