# MERCHANT_LOW_TECH_ONBOARDING_DESIGN_V1

## 1. 商家低技术能力假设

默认商家不需要理解以下技术词，也不应在界面中直接暴露这些词：

- 微信开放平台
- authorizer_access_token
- card_id
- cardExt
- API 回调
- 核销接口

商家的真实诉求是：

- 少操作
- 看得懂
- 照着做就能完成接入
- 出问题时知道找谁

## 2. 商家端最简流程

第一步：填写店铺基础信息  
第二步：选择卡券模板  
第三步：确认活动参与  
第四步：扫码授权公众号  
第五步：添加核销员  
第六步：查看审核结果

## 3. 平台代办流程

平台承担更多自动化与代办动作，商家只做关键确认：

- 平台生成卡券草稿
- 平台生成素材建议
- 平台检查门店状态
- 平台检查核销员状态
- 平台生成授权二维码
- 平台跟踪微信审核
- 平台生成培训材料

## 4. 商家界面设计原则

界面中禁止展示技术词，统一替换为商家能理解的表达。

禁止展示：

- token
- card_id
- API
- 回调
- 开放平台
- 权限集

统一替换为：

- 授权状态
- 卡券状态
- 门店状态
- 核销员状态
- 审核状态

## 5. 自动化检查清单

接入前，系统自动或平台代办检查以下项目：

- 公众号是否认证
- 卡券功能是否开通
- 门店是否审核通过
- 是否已授权 AR游伴
- 是否已配置核销员
- 卡券是否审核通过
- 是否可发放
- 是否可核销

## 6. 平台运营后台功能建议

平台运营侧建议提供以下能力：

- 商家接入进度看板
- 一键生成卡券方案
- 一键生成授权链接
- 一键生成商家 SOP
- 一键生成店员核销说明
- 异常提醒

## 7. 首场活动最低可用方案

首场活动的商家侧最低可用方案只要求商家完成：

- 扫码授权
- 确认卡券内容
- 添加核销员
- 店员培训

其余环节尽量由平台代办或自动检查完成。

## 8. 设计结论

本期商家接入设计的核心原则是：

- 商家少操作
- 平台多代办
- 系统多自动检查
- 流程必须简单
- 术语必须商家能懂
