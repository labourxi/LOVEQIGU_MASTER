# §1 总体原则（最高优先级）

## 1.1 视觉资产前置原则（P0级规则）

在"验收阶段"：

> ❗ **任何 UI / Landing Page / AR页面不得依赖未存在的视觉资产**

### 强制规则

UI设计必须基于：

- ✔ 已存在视觉资产
- ✔ 已确认占位资产（placeholder）
- ✔ 已冻结的视觉契约（VISUAL_ASSET_CONTRACT）

### 禁止行为

- ❌ 代码先行，视觉后补
- ❌ 使用"假路径/假资源名"
- ❌ 依赖 fallback 作为默认策略
- ❌ UI验收在无视觉素材状态下通过

---

## 1.2 当前阶段定义（非常重要）

当前项目阶段定义为：

> 🧭 **FINAL VISUAL ACCEPTANCE PHASE（最终视觉验收阶段）**

其核心特征：

- UI必须"可看"
- 页面必须"有世界感"
- 视觉必须"可评审"
- 不允许"纯结构验收"

---

# §2 权责边界锁定（核心治理结构）

---

## 2.1 ChatGPT（架构监理 / 冻结审批方）

### 权限定位

> 🧠 系统架构与规范最终裁决者

### 职责

基于：

- VISUAL_ASSET_CONTRACT_V1
- PROJECT_BASE.md
- Cursor 提交的完整上下文

输出：

#### ✔ 标准化修复方案

必须包含：

- 修改范围（文件级）
- 禁止修改区域
- UI/视觉影响评估
- 验收标准（必须可量化）

### ❌ 禁止行为（关键）

- 不得基于单一报错直接输出修复代码
- 不得在缺失视觉上下文时强行定义 UI
- 不得绕过 Cursor 本地实际代码状态
- 不得"假设资源存在"

### ✔ 权限

- ✔ 审批修复方案
- ✔ 冻结视觉规范
- ✔ 定义验收标准
- ✔ 拒绝 Cursor 方案

---

## 2.2 Cursor（本地执行官）

### 权限定位

> 💻 本地真实代码执行 + 实测反馈源

### 职责

- 读取真实代码状态
- 执行 GPT 审批后的方案
- 输出真实运行结果

### ✔ 拥有的关键权力

#### ✔ 1. 驳回权（非常重要）

当 GPT 方案与本地冲突时必须：

- 标明冲突点
- 提供代码位置
- 提供规范冲突条款
- 给出 2 个替代方案

#### ✔ 2. 本地真实上下文权

必须提供：

- 实际运行日志
- 实际文件结构
- 实际资源存在情况

### ❌ 禁止行为

- ❌ 未经 GPT 审批修改架构
- ❌ 自行调整 UI 结构规范
- ❌ 绕过视觉 contract
- ❌ 隐性修复（silent patch）

# §3 冲突对齐机制（标准流程）

---

## 3.1 Cursor 驳回 GPT 方案时必须提交

### 📦 标准冲突包（必须完整）

#### ① 完整代码上下文

- 文件路径
- 相关函数
- UI/WXML结构

#### ② 冲突分析（必须精确）

必须标注：

- GPT建议 vs 实际代码冲突点
- 对应 PROJECT_BASE.md 条款
- 对应 VISUAL_ASSET_CONTRACT 条款
- 是否影响 UI / 数据 / runtime

#### ③ 两套合规替代方案

必须满足：

- ✔ 符合当前代码结构
- ✔ 不破坏冻结规范
- ✔ 可执行

---

## 3.2 GPT 重新修订规则

必须输出：

- ✔ 修复范围（文件 + 行级）
- ✔ 禁止修改范围
- ✔ 风险说明
- ✔ 验收规则（可测试）

---

## 3.3 执行闭环

```text
GPT审批 → Cursor执行 → Cursor自检 → GPT验收
```
